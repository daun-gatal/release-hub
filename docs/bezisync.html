What it is

A control plane for Change Data Capture you drive from a web UI. It orchestrates Debezium Server — one container per pipeline — so each stream has its own runtime, config, state, logs, and health model.

From database to trusted stream

BeziSync turns the usual CDC setup into a guided path with checks at the points where Debezium commonly fails.

1 Add secrets

Store source and broker credentials once. BeziSync encrypts them and reuses them across connections.

2 Test connections

Verify Postgres/MySQL CDC settings and Kafka reachability before a pipeline is created.

3 Pick tables

Choose schemas, tables, and snapshot behavior in the pipeline wizard.

4 Validate & deploy

Preview topics and generated properties, then launch a dedicated Debezium Server container.

5 Observe & inspect

Watch phase, metrics, logs, and recent Kafka records from the pipeline cockpit.

6 Operate safely

Restart, stop, backfill with signals, or edit config only when the pipeline state allows it.

What makes it useful day to day

BeziSync is built around the operator moments that decide whether a CDC pipeline feels reliable: setup, first deploy, debugging, proof of data, and controlled change.

Readiness

Source checks that know CDC

Postgres checks cover logical WAL, replication slots, WAL senders, replication privilege, table discovery, and primary-key or replica-identity warnings. MySQL checks cover binary logging, row binlogs, server id, grants, GTID hints, and keyability.

Compiler

Generated Debezium properties

The compiler injects connector class, topic prefix, Postgres slot/publication, MySQL server id, serializers, offset storage, schema history, heartbeat, formats, table filters, and transforms.

Transforms

Friendly SMT catalog

Add common single-message transforms without memorizing class names: unwrap Debezium envelopes, route topics, mask sensitive fields, or drop fields. Advanced custom SMT classes still work.

Preview

See the data that landed

Open the Data Preview tab to read recent Kafka records for a pipeline topic, including operation, table, key, before/after payload, partition, offset, and timestamp.

Backfill

Incremental snapshots without redeploy

Use Debezium source-table signals to execute, pause, resume, or stop incremental snapshots while the pipeline keeps streaming.

Runtime

Deploy where your streams should run

Each workspace can target a Docker daemon, TLS cert path, network, Debezium image, resource limits, restart policy, and metrics settings from Runtime Config.

Why teams try it

Debezium is powerful but unforgiving. BeziSync turns each common failure mode into something caught early, explained clearly, and visible in the UI.

Manual Debezium setupWith BeziSync
Cryptic failures after deploy Preflight validation — connects to source/target and checks wal_level, replication slots & privileges (Postgres), log_bin/binlog_format/grants (MySQL), and broker reachability. Blocks bad deploys with actionable remediation.
Hand-written connector properties Connector-aware compiler injects the right connector class, topic prefix, Postgres plugin/slot/publication, MySQL server.id, and offset & schema-history storage — and shows you the result before deploy.
"Is it even working?" Live health — lifecycle phase, snapshot completion, throughput hints, parsed error reason, and a live log viewer.
Slow, error-prone first run A guided wizard (configure → select tables → validate → deploy) so the pipeline is only created once everything lines up.

Key concepts

Four building blocks. You set them up once, then compose them into as many pipelines as you need.

🔑

Credentials

Stored secrets for your databases and brokers — username/password, SASL, AWS IAM, or none. Encrypted at rest and reused across connections.

🔌

Connections

A source (PostgreSQL or MySQL) or a target (Kafka). Each can be tested on the spot; sources are verified for CDC readiness.

🔄

Pipeline

One Debezium stream binding a source to a target, with the tables you picked and a snapshot mode. Runs as its own Debezium Server container.

🗂️

Workspace

Groups credentials, connections, and pipelines so teams and environments stay separated.

Run it yourself

Start with the compact self-hosted stack, then point BeziSync at the Docker daemon that should run your Debezium containers. Bring your own source database and Kafka, or use the production-shaped Docker-in-Docker example for a fuller local simulation.

Save this as docker-compose.yml:

# docker-compose.yml — BeziSync self-host (bring your own source DB + Kafka)
services:
  db:                  # BeziSync's own metadata store
    image: postgres:16
    environment:
      POSTGRES_USER: bezisync
      POSTGRES_PASSWORD: password
      POSTGRES_DB: bezisync
    volumes: [metadata:/var/lib/postgresql/data]

  redis:               # Celery broker for the worker
    image: redis:7

  init:                # creates schema + seeds the admin user, then exits
    image: ghcr.io/daun-gatal/bezisync-backend:latest
    command: python -m bezisync.scripts.init_db
    environment: &env
      DATABASE_URL: postgresql+psycopg2://bezisync:password@db:5432/bezisync
      REDIS_URL: redis://redis:6379/0
      BEZISYNC_ENCRYPTION_KEY: ${BEZISYNC_ENCRYPTION_KEY:?generate a Fernet key}
      BEZISYNC_JWT_SECRET: ${BEZISYNC_JWT_SECRET:?set a random secret}
      BEZISYNC_ADMIN_PASSWORD: ${BEZISYNC_ADMIN_PASSWORD:?set an admin password}
    depends_on: [db]

  api:
    image: ghcr.io/daun-gatal/bezisync-backend:latest
    command: uvicorn bezisync.main:app --host 0.0.0.0 --port 8000
    environment:
      <<: *env
      BEZISYNC_CORS_ORIGINS: '["http://localhost:8080"]'
    ports: ["8000:8000"]
    depends_on: [init, redis]

  worker:              # spawns/stops the per-pipeline Debezium containers
    image: ghcr.io/daun-gatal/bezisync-backend:latest
    command: celery -A bezisync.worker.celery_app worker --beat --loglevel=info
    environment: *env
    depends_on: [init, redis]

  frontend:            # the web UI
    image: ghcr.io/daun-gatal/bezisync-frontend:latest
    ports: ["8080:80"]
    depends_on: [api]

volumes:
  metadata:

Provide the three required secrets, then start the stack:

# a one-liner to generate a Fernet encryption key:
export BEZISYNC_ENCRYPTION_KEY=$(python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())")
export BEZISYNC_JWT_SECRET=$(openssl rand -hex 32)
export BEZISYNC_ADMIN_PASSWORD=<your-admin-password>

docker compose pull
docker compose up -d
ServiceURL
Web UIhttp://localhost:8080
APIhttp://localhost:8000
One-time setup after first start: the worker deploys Debezium connectors to a Docker daemon you choose — it isn't wired up by default. Log in, then open Settings → Runtime Config and point BeziSync at the daemon that should host your connectors (and its image, network, and TLS). Until that's set, pipelines can be created and validated but not deployed.
Advanced: simulate the production runtime with Docker-in-Docker

For a closer production-style test, run BeziSync with a dedicated Docker daemon instead of your host Docker socket. Download the Docker-in-Docker compose example alongside its source init SQL and change generator, then set Runtime Config to:

Docker host: tcp://dind:2376
TLS verify: on
Cert path: /certs/client
Docker network: leave blank

The example includes a CDC-ready Postgres source, Kafka target, continuous row changes, and a separate mTLS Docker daemon where BeziSync launches Debezium containers.

How you use it

Everything happens in the BeziSync UI. From a clean workspace, a useful CDC stream is a guided path rather than a properties-file guessing game.

  1. Add credentialsStore the secrets for your source database and target broker. They're encrypted immediately and never shown again.
  2. Create connectionsAdd a source (Postgres/MySQL) and a target (Kafka), then hit Test. BeziSync confirms connectivity and that the source is CDC-ready.
  3. Launch a pipeline (the wizard) Configure — name the stream, bind source → target, pick a snapshot mode.
    Select tables — discover tables from the source and choose which become topics (or add schema.table by hand).
    Validate & deploy — preflight runs every check, previews the topics and the compiled Debezium config, then deploys. Failing checks block the deploy; warnings can be overridden with Deploy anyway.
  4. Monitor & operateThe pipeline page shows lifecycle phase, snapshot progress, lag, queue usage, uptime, parsed errors, and live logs. Trigger ad-hoc incremental snapshots (execute / pause / resume / stop) on a running pipeline — no redeploy needed.
  5. Preview & refineUse the Data Preview tab to inspect recent Kafka records, then refine filters, formats, heartbeat, signaling, or SMT transforms when the pipeline is stopped.

Snapshot modes

Chosen per pipeline in the wizard's Configure step — it decides what happens with existing rows before streaming begins.

ModeBehavior
Initial (recommended)Snapshot existing rows once, then stream ongoing changes.
Schema onlySkip existing rows; stream changes from now on.
NeverStreaming only — no snapshot.
AlwaysRe-snapshot on every start.
Already running and need to backfill a table? Use an incremental snapshot from the pipeline page instead of redeploying — it runs alongside live streaming and can be paused, resumed, or stopped.

Under the hood

A small app stack manages metadata and orchestration. Your actual CDC traffic flows through Debezium Server containers into your Kafka cluster.

Web UI

Where you log in, manage credentials and connections, run the pipeline wizard, and watch live monitoring.

API + worker

The API handles connectivity checks, preflight validation, config compilation, Kafka preview, signaling, and health parsing. A background worker spawns, reconciles, and stops the per-pipeline Debezium Server containers.

Metadata services

Postgres stores BeziSync metadata, Redis backs the worker queue, and encrypted credentials stay inside the BeziSync control plane.

Your infrastructure

Source databases, Kafka, Docker runtime, networks, and TLS settings stay under your control.

FAQ

What is the quickest way to see it working?

Run the compose stack, log in, open the seeded orders-to-kafka pipeline, deploy it, then watch health and preview records in Kafka.

Do I need Kafka Connect?

No. BeziSync orchestrates Debezium Server containers directly, with one container per pipeline.

What source and target systems are supported?

BeziSync currently focuses on PostgreSQL and MySQL sources with Kafka as the target.

Does BeziSync store my table data?

No. It stores app metadata, connection definitions, encrypted credentials, and pipeline state. Row changes stream through Debezium to Kafka.
